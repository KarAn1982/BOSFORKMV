# БОСФОР — спецификация форм, CRM и защиты заявок

Дата: 19 июня 2026 года

## Цель

Передавать B2C- и B2B-заявки в CRM без потери источника, согласия и приложенных
проектных файлов. Персональные данные не должны попадать в системы аналитики.

## Единая точка приёма

`POST /api/leads`

Формат: `multipart/form-data` для форм с файлами, JSON допустим для коротких форм.

Успешный ответ:

```json
{
  "ok": true,
  "lead_id": "crm-or-internal-id",
  "message": "Заявка принята"
}
```

Ошибки:

- `400` — некорректные или неполные данные;
- `413` — превышен размер файла;
- `415` — запрещённый тип файла;
- `422` — не подтверждено согласие;
- `429` — превышен лимит отправок;
- `500/502/503` — заявка сохранена в очереди или временно не принята.

## Поля лида

| Поле | Назначение |
|---|---|
| `form_id` | Стабильный идентификатор формы |
| `audience` | `b2c`, `developer`, `architect`, `corporate` |
| `name` | Имя контактного лица |
| `phone` | Телефон в нормализованном формате |
| `email` | Email |
| `company` | Компания для B2B |
| `object_type` | Тип объекта |
| `region` | Регион объекта |
| `message` | Задача, стадия проекта, комментарий |
| `consent` | Факт согласия |
| `consent_version` | Версия текста согласия в формате YYYY-MM-DD |
| `page_url` | Страница отправки |
| `referrer` | Предыдущая страница |
| `utm_*` | UTM-метки |
| `yclid`, `gclid` | Рекламные идентификаторы при наличии |
| `client_id` | Обезличенный идентификатор аналитики |
| `files[]` | Проектные документы |

Телефон или email обязателен. Для B2B желательно требовать компанию, тип объекта и
описание стадии проекта.

## Маршрутизация в CRM

| Аудитория | Воронка | Ответственный | SLA первого контакта |
|---|---|---|---|
| B2C | Частные клиенты | Менеджер розничных продаж | до 15 минут в рабочее время |
| Застройщик | Корпоративные проекты | Руководитель B2B-продаж | до 2 рабочих часов |
| Архитектор | Инженерное сопровождение | Инженер проекта | до 4 рабочих часов |
| Корпоративный заказчик | Корпоративные проекты | B2B-менеджер | до 2 рабочих часов |

Конкретная CRM, идентификаторы воронок и ответственные подтверждаются владельцем
процесса до разработки интеграции.

## Файлы

- Разрешены: PDF, DWG, DXF, DOC, DOCX, XLS, XLSX, ZIP.
- Лимит прототипа: 20 МБ на файл.
- Сервер проверяет расширение, MIME и сигнатуру файла.
- Имя файла заменяется безопасным внутренним идентификатором.
- Архивы проверяются на вложенные исполняемые файлы и архивные бомбы.
- Файлы проходят антивирусную проверку до передачи ссылки сотруднику.
- Хранение — закрытое объектное хранилище; CRM получает временную подписанную ссылку.
- Срок хранения и удаление определяются политикой обработки персональных данных.

## Антиспам

Обязательная серверная цепочка:

1. Honeypot-поле и минимальное время заполнения.
2. Rate limit по IP и fingerprint без создания постоянного рекламного профиля.
3. Проверка Origin/Referer и CSRF для same-site отправки.
4. Turnstile или аналог только после подозрительного поведения, без капчи по умолчанию.
5. Дедупликация по телефону/email и `Idempotency-Key`.
6. Очередь повторной доставки при недоступности CRM.
7. Журнал результата без сохранения текста заявки в обычные application-логи.

## Аналитика

До отправки:

- `form_start`;
- `file_upload`;
- `file_upload_rejected`.

После ответа сервера:

- `lead_submit_success`: `form_id`, `audience`, `lead_id`, `response_time_ms`;
- `lead_submit_error`: `form_id`, `audience`, `error_type`;
- `crm_delivery_success` и `crm_delivery_error` — только в серверном мониторинге.

Телефон, email, имя, текст заявки и имена файлов в аналитику не передаются.

## Согласие и прозрачность

- Рядом с кнопкой размещается ссылка на политику и текст согласия.
- Сервер сохраняет версию текста согласия, дату, время и источник.
- Маркетинговая подписка не объединяется с обязательным согласием на обработку заявки.
- Юрист подтверждает реквизиты оператора, сроки хранения и трансграничную передачу.

## Критерии приёмки

- Валидная заявка создаётся в правильной CRM-воронке не более чем за 30 секунд.
- Повтор одной заявки не создаёт дубль.
- При недоступности CRM лид сохраняется в очереди и доставляется повторно.
- Запрещённые и слишком большие файлы не попадают в хранилище.
- Ошибка понятна пользователю и не очищает введённые поля.
- Событие успеха отправляется только после подтверждения сервера.
- Тестовые заявки имеют отдельную метку и удаляются после приёмки.

## Исполняемый прототип

`prototype_server.js` реализует локальный `/api/leads` и проверяется
`test_prototype_api.js`. Он предназначен для end-to-end приёмки интерфейса,
валидации и аналитики. Реальная CRM, постоянная зашифрованная очередь,
антивирус и объектное хранилище остаются production-интеграциями.

Transport-neutral delivery layer реализован в `deployment/crm-delivery.js`.
Маршрутизация, retry, idempotency и dead-letter проверяются mock CRM-тестом.
Production mapping и durable queue описаны в `BOSFOR_CRM_DELIVERY_IMPLEMENTATION.md`.

ZIP central directory проверяется до enqueue: entry count, uncompressed size,
compression ratio, encryption, path traversal и executable entries. Лимиты и
production quarantine-граница: `BOSFOR_ARCHIVE_SECURITY.md`.

Server schema validation использует allowlist реальных полей 32 форм, отклоняет
неизвестные поля, ограничивает UTM/ad IDs до 200 символов, URL до 2048 и
разрешает referrer только с HTTP(S)-схемой.
